Java Coding Standards
Last Updated: August 12, 2017

Java Language Requirements

  1. Every source file in Java defines one class. If the name of class is ClassName, then the source file must be ClassName.java. Like C, Java is case sensitive.

  2. Java uses packages to break up the functionality of larger programs. Packages have a hierarchical naming convention. For instance, packages created by CPE programmers should be named as follows: 

            th.ac.kmutt.cpe.MyPackage
	th.ac.kmutt.cpe.MyPackage.Utils
    If you use packages, Java requires that you keep your source files in a directory structure that matches the hierarchical package structure. For example, all the source files for MyPackage in the above example must be located in: 
        [devroot]/th/ac/kmutt/cpe/MyPackage
    The directory [devroot] can be any directory. This is where you must be located when you run the program.

Comments

  1. Java accepts, and you may use, both C and C++ style comments. That is, you can use either /* .... */ or // for inline comments. However, all header comments for classes, methods and variables must use the comment style required by the javadoc utility, that is, they must begin with /** (two asterisks) and end with */.

  2. Immediately before the class declaration in each Java source file, you must have a comment that includes the name of the class, a one-sentence description of what the class does, the programmer's name and nickname, the programmer's student ID, and the creation date. Note that this comment must begin with /** not /* so that it will be recognized by the javadoc utility.

    Example

     /**
  *  
  * EmailMessage
  *
  *   This class represents a single message, either created or
  *   received, in an email client application.
  *
  *   Created by Suwannee Promprasong (Nee)  ID 58070703435
  *     6 Sept 2017
  *
  *
  */  
public class EmailMessage
{
   ...
}

    The only thing before this comment should be any import statements required by your class.

  3. Immediately before each method (function), you must write a comment beginning with /** that explains what the method does, lists and explains the function arguments, and explains the return value if any. You must use the javadoc prefixes @param and @return to identify the different kinds of information.

    Example

        /** 
     * Return one of the X coordinates
     * @param  which    1, 2, or 3 specifying which vertex we want
     * @return vertex X coordinate, or -1 if argument is not 1, 2 or 3
     */
    public int getX(int which)
    {
    ....
    }

  4. Immediately befor the declaration of each member data item, you must write a comment starting with /** that explains what the member data means and what it is used for.

    Example

       /** X coordinates of three points defining the triangle */
   private int xcoord[] = new int[3];

   /** Y coordinates of three points defining the triangle */
   private int ycoord[] = new int[3];

  5. Comments should be used elsewhere to introduce sections of the program or to explain calculations that might not be clear to the reader. However, you should not put in comments where the purpose is obvious. Excessive use of comments makes a program harder to understand, not easier!

    Examples

           /* Initialize Vat to zero so we do not need an else clause*/        
       vat = 0;
       
       ....

       /* area formula is  
         |Ax(By - Cy) + Bx(Cy - Ay) + Cx(Ay - By)|/2
       */
       int numerator = xcoord[0] * (ycoord[1] - ycoord[2]);
       numerator += xcoord[1] * (ycoord[2] - ycoord[0]);
       numerator += xcoord[2] * (ycoord[0] - ycoord[1]);

  6. The declaration of each non-member variable must include a short comment explaining the purpose or meaning of the variable. Exception: if you use i, j as loop indices, no comment is necessary.

    Example

            int totalOnes;  /* count of one baht coins seen */
        int totalFives; /* count of five baht coins seen */
        String input;   /* space to store text typed by user */

  7. If you use arrays, your variable comment must also explain the use of each subscript

    Example

            int population[77];              /* population of each of the 77 Thai provinces */
	String provinceNames[77];        /* names of the corresponding provinces */
    As shown above, it is nice if your variable comments line up neatly, so that they are easy to read.

Format and Indenting

  1. The curly braces that enclose the class definition should be at the start of the line. Do not indent them:

    Example

    public class Triangle
{
   /* code.... */
}

  2. The declaration for a method should be indented four spaces from the left edge of the page.

  3. The curly braces for a method should also be indented four spaces from the left edge of the page, and should line up vertically with the method declaration.

  4. The statements in a method should be indented four spaces from the starting brace of the function, or from the starting brace of their controlling statement if they are in a compound statement block.

  5. Comments should be indented so they line up with the statements.

    Example of correct indenting for methods

        /**
     * calculate the perimeter of this triangle
     * @return perimeter value
     */
    public double calcPerimeter()
    {
	double perimeter = 0;
        /* loop, summing up the length of the three sides */
        for (int i = 1; i < 4; i++)
	{
            /* calcLength returns the length of one side */			    
	    perimeter = perimeter + calcLength(i);
	} 
        return perimeter;
    }

  6. All switch statements should be indented like the following:

    Example

        switch(value)
    {
        case 1:
            printf("This is case 1\n");
            break;
        case 2:
            printf("This is case 2\n");
            break;
        default:
            printf("NO MATCH!\n"); 
    }
    Do not put the break on the same line as any other statement. Do not put anything after the colon in the case line.

  7. You must never use so-called "running dog" braces in code that you write for my class. "Running dog" means that the starting brace of a block comes at the end of the previous line rather than on a line of its own. Programs that use this style are very difficult to understand because it is hard to match up the beginning and end of blocks. This is a common style used by some Java programmers and Java editors, but I will not accept it.

    BAD Example

       /* DO NOT FORMAT LIKE THIS! */
   if (format.compareTo("Running Dog") == 0) {
      System.out.println("Serious coding standards error!\n");
      System.out.println("You will lose credit! \n"); 
      }

Names

  1. Do not use underscores to separate words in any names in your Java programs.

  2. Class names must begin with an upper case letter. If the class name is multiple words, each word should be capitalized.

  3. Names of methods must begin with a lower case letter. If the method name is multiple words, each word after the first should be capitalized.

  4. Names of variables and member data items must begin with a lower case letter. If the name is multiple words, each word after the first should be capitalized.

    Examples

    public class EmailMessage
{
}
...
  int initialPopulationValue = 0;
  String provinceName; 
  double percent = 0.0;   
  double calculatePopulation(int thisWeek, 
                             double changeRate, 
                             int weekCount);

  5. Names of variables should be meaningful and express what the variable does.

  6. Names of methods should be meaningful and express what the method does.

  7. Avoid using abbreviations in variable and method names except when the meaning of the abbreviation is very commonly understood. 

       int stdCount;        /* Not a good name */
   int studentCount;    /* Much better */
   float vat;           /* Ok because this is a common abbreviation */
   float gpa;           /* Ok because this is a common abbreviation */

Program Structure

  1. If your class includes a constructor, this should be the first defined method of the class.

  2. If your class includes a main() method, this should be the last defined method of the class.

  3. Group your methods according to what they do. For instance, if you have multiple getter and setter methods, define them together rather than interspersing the definitions of other methods.
